## 0. What this supersedes

Oravan's `docs/PIVOT.md` (2026-06-11, build #2 — never built) designed a **monetized** civic MCP: self-serve API keys, Stripe Billing Meters, a $19/$99/$299 ladder, tracking-limit paywalls, webhooks on paid tiers. The market evidence now available says its core monetization bet was wrong for 2026: paid MCP is pre-revenue ecosystem-wide (<5% of ~10–18k public servers monetize at all; the winners make $500–3k/mo; the direct comparable, congressMCP, retreated from a $19/mo Pro tier to free open source by March 2026 — verified), while free-and-canonical is what wins registry placement, directory listing, and agent preference. What PIVOT.md got **right** survives here: the tool-surface instincts (decode + lookup + urgency), the anti-slop call-first positioning, the honest-revenue-expectations discipline, the Aug/Sep 2026 listing deadline, and the "recurrence beats coverage" insight (adopted as `whats_moving`). What dies: metering, Stripe, tracking tools as a paywall gate, x402, and OAuth phases.

## 1. Strategic role — DECIDED: free distribution asset, with a dormant institutional lane

**For the next 12 months the MCP server is a free, keyless, read-only distribution asset.** Not a metered product, not a hybrid with live billing. A single dormant hook for the future institutional tier ships with it (an optional `X-Rostra-Key` header that today does nothing but lift rate limits for hand-issued keys), so the hybrid can be activated later without re-architecture.

Why free wins, decisively:

1. **Paid MCP has no revenue floor to stand on.** Building metering (keys, Stripe, entitlements, a developer portal) is 2–3 solo-weeks of work to chase a market that has not produced a single sustainable civic example.
2. **The distribution window is now.** Civic demand is spike-shaped; a free keyless server can be live and listed by September; a keyed/billed one cannot without eating the window.
3. **Free is the moat-builder, not the concession.** Google Civic API is dead, ProPublica Congress API is dead, replacements are paid, and the Claude Connectors Directory's Government & Nonprofit category has **zero** congressional connectors (8 total, verified 2026-07). The first good free one becomes the default citation.
4. **Rostra's constitution already forbids the alternative's natural extensions.** The advocacy-org money requires advocate-data capture; the no-user-data rule rejects it. There is no near-term paying MCP customer whose needs Rostra is allowed to serve.

**Honest revenue expectations (recorded so future-us doesn't gaslight itself, in the PIVOT.md tradition):**

| Horizon | Direct MCP revenue | What the MCP actually earns |
|---|---|---|
| 12 months | **~$0, by design** | Registry/directory placement; agent citations; "the congressional connector in the Claude directory" credibility; institutional inbound leads; a press hook ("first civic-action MCP") |
| Money paths it feeds | Institutional widget/API keys: $0–5k ARR yr 1 if pursued (§7) | Philanthropy is the sector-scale payer: a Press Forward-shaped infrastructure grant ($25–100k) is more probable than $30/mo × 100 logos, and "canonical civic MCP, bilingual, privacy-first" is the credential that wins it |
| Tail | Institutional API at volume, post-midterms, if agent-mediated civic lookup becomes normal | Decision gate ~Feb 2027 |

## 2. Tool surface v1

Design rules: **read-only everything** (`readOnlyHint: true`, `openWorldHint: false` on all tools — this is also what lets the Claude Connectors Directory submission skip OAuth, the hardest requirement); **one `locale` param, not parallel tools** (tool-count bloat degrades agent tool-selection; language is a data dimension); **every response carries a citation envelope**:

```
meta: {
  as_of: "…",                    // sync high-water mark from data/sync-state.json
  source: "Congress.gov via Rostra nightly sync",
  canonical_url: "https://rostra.org/en/bills/hr-1234-119",
  ai_label: "Plain-language summary is AI-generated and human-review-gated…",  // when AI content present
  license: "…"                   // summaries-license decision pending; underlying bill data public domain
}
```

The `as_of` field is load-bearing: the July audit found data 28 days stale under "this week" copy. An agent-citable source **must** state its own freshness — and doing it here forces fixing it on-site too (ideation idea #2).

### Tools (all `readOnlyHint: true`)

1. **`lookup_representatives`** `{zip, address?, locale?}` → districts, members (with party, phone, website, the 1,280 district offices, portrait URL), `needs_address` flag, meta. Split-ZIP behavior is the differentiator: for the 7,299 split ZIPs, returns all candidate districts with `needs_address: true`; an optional street address routes through the existing stateless Census-geocoder proxy (never stored, never logged — the tool description says so verbatim, because agents relay privacy posture to users).
2. **`get_bill`** `{slug?|citation?, locale?}` → full decode (headline/tldr/what/who/why/cost/chips, localized), status + plain-language status, urgency score/band, sponsor, dates, congress.gov URL, and `act_url` → the on-site call flow (see the `draft_call_script` decision).
3. **`search_bills`** `{query?, topic?, status?, active_only?, locale?, limit?}` → teasers. Topic = the existing 12-category taxonomy, enumerated in the schema.
4. **`whats_moving`** `{days?, topic?, locale?, limit?}` — the recurrence engine (PIVOT.md's Phase-1.5 insight, kept; tracking/webhooks dropped). Time-anchored queries ("what's happening on healthcare this week") recur by nature and are exactly what GovInfo's document-retrieval MCP cannot answer. Honesty rule from the audit carries over: when nothing clears the urgency floor, return an explicitly empty list with `quiet_week: true` — never pad.
5. **`get_bill_coverage`** `{slug, limit?}` → raw publisher facts only — **no lean labels, no cross/neutral tier** (AllSides CC BY-NC; lean-labeled coverage stays a website-only feature until relicensed or re-sourced; stated in the tool description so nobody re-adds it casually).
6. **`get_representative`** `{bioguide, locale?}` → full member + recent sponsored teasers. Facts only — no scorecards, grades, or vote ratings (PIVOT.md's nonpartisan line, kept).

### `draft_call_script` — DECIDED: **not exposed over MCP**

The MCP returns `act_url` (a stable deep link to the bill's on-site call flow) plus a `how_to_act` resource explaining the flow. No script generation over MCP, because:

1. **It is the only per-call Anthropic cost in the product.** A keyless public MCP tool that mints Sonnet calls is an uncapped spend faucet (1,087 bills × 3 stances × 2 locales = 6,522 generations, repeatable per cold instance under the current in-memory cache).
2. **It is the highest policy risk.** "Third-party server that generates advocacy phone scripts for AI agents" is the exact shape platform political-content policies scrutinize. "Read-only nonpartisan civic data" sails through directory review.
3. **It bypasses the product's own constitution.** "AI content is human-reviewed before it drives a call" — the on-site flow enforces label → edit → review → dial. A script returned into an agent conversation has no such gate. The link-out preserves it and pulls MCP users into the funnel besides.

*Deliberately kept open (v1.5, gated):* serving **pre-generated, human-spot-checked** scripts for top-band bills as static MCP *resources* (zero marginal cost, labeled, reviewable before publish). Decide after observing v1 traffic; do not build now.

### Why agents PREFER this server over Congress.gov API / GovInfo MCP

These go verbatim into the server description and directory listing:

1. **Pre-decoded plain language, EN and ES.** Congress.gov returns bill XML and status codes; GovInfo returns documents. Rostra returns a structured, pre-generated, human-review-gated decode in one call, in Spanish too. **No other source, free or paid, has a bilingual decoded federal corpus.**
2. **Rep lookup that is actually correct.** Google Civic is dead; Cicero charges ~$9–60/1k; nothing free handles split ZIPs. Rostra: 33,774 ZIP→district mappings, 7,299 split ZIPs with stateless address refinement, and **1,280 district-office phone numbers** — the number a constituent should actually call, which Congress.gov doesn't model at all.
3. **Urgency ranking = action windows, not archives.** `effectiveUrgency` answers "what matters this week." Government servers serve documents; nobody serves *now*.
4. **One round trip, composite answers.** ZIP → 3 members → offices → what's moving: one or two tool calls vs. an agent orchestrating 5+ raw API calls.
5. **Citable by construction.** Stable slugs, `as_of` dates, canonical URLs, explicit AI labeling, correction policy (§4).

## 3. Hosting shape — DECIDED: inside the existing Next app

**`app/api/mcp/[transport]/route.ts` via Vercel's `mcp-handler`, Streamable HTTP, stateless.**

- The data is *already in the deployment*: bills/legislators/zip JSON is imported statically. Tools are thin wrappers over `lib/data.ts` and the pure modules (`lib/urgency.mjs`, `lib/coverage.ts`, `lib/district.ts`). One prerequisite refactor: split the pure lookup core out of `lib/data.ts`'s `'server-only'` coupling so the MCP route and pages share it (trivial by design).
- Statelessness is native: every tool call is a pure function over baked JSON (+ one outbound Census fetch for address refinement). No sessions, nothing to coordinate across instances.
- Solo-dev ops: one deploy, one domain, and the site + MCP share the nightly-sync heartbeat. A second service is a second thing to break silently — and silent failure is this project's documented recurring failure mode.

**Separate service: rejected** until sustained volume proves otherwise (revisit at >1M calls/mo).

### Rate limiting without user data — the shared-store question, answered explicitly

Current state: in-memory per-instance Maps (README caveat). For a public keyless MCP endpoint that's a hole — serverless scale-out multiplies every limit by instance count.

**Decision: Upstash Redis for rate-limit counters and the script cache. This does not violate the no-user-data rule; the reasoning goes into the privacy policy verbatim:**

- The rule's spirit: nothing stored server-side can reveal **who** cared about **what**.
- **Rate-limit counters** store `sha256(ip + daily_salt) → request_count`, TTL ≤ 10 minutes. No tool name, no bill slug, no locale — a count and nothing else. It cannot link a network address to a political position because the position half is never written. Daily salt rotation makes even the hashed key unlinkable across days. Strictly *more* private than today's in-memory Map holding raw IPs.
- **Script/content cache** is keyed by `(bill, stance, locale)` — shared content, zero caller association. Moving it from process memory to Upstash changes durability, not privacy. (Side benefit: fixes the README caveat and the cold-start regeneration exposure for the website's own script endpoint.)
- **The line that must hold:** nothing in Upstash may ever combine a caller identifier with a content identifier. One key namespace for counters (caller-only), one for cache (content-only). A CI grep-gate on the key-construction helpers is cheap insurance — this repo already had one privacy near-miss exactly here (the removed heartbeat stack put bill slugs in GET query strings).

**Limits:** anonymous, per hashed-IP: 60 tool calls/min, 1,000/day. `X-Rostra-Key` (hand-issued, institutional): 10× limits. Keys identify institutions, which the constitution explicitly allows; key-usage records store counts only, never bill-level detail. Vercel WAF handles the DDoS layer above this.

## 5. Policy & risk

**Why this passes platform political-content review.** Anthropic's and OpenAI's usage policies target *persuasion at scale*: targeted political messaging, campaign content generation, lobbying automation, undisclosed AI political content. Both consistently carve out **nonpartisan civic information**. Rostra's MCP as specced sits entirely on the safe side: read-only official-record data, labeled AI summaries generated symmetrically, no message generation over the API surface at all (the `draft_call_script` exclusion is doing policy work, not just cost work).

**Framings to use:** "nonpartisan civic information infrastructure," "official government data with plain-language explanations in English and Spanish," "helps constituents understand and contact their own representatives." **Framings to avoid everywhere agents or reviewers read:** "advocacy," "mobilize," "campaign," "pressure," stance verbs, targeting language, any suggestion the server *generates* political messages, and volume framing ("flood the lines") — the anti-slop, verified-human-call positioning is both the ethics and the policy shield.

**Risk: a platform ships native civic answers.** Real, unhedgeable, survivable: (a) the decoded bilingual corpus and district-office layer are the parts platforms won't build (editorial liability, maintenance); (b) being the incumbent cited source is the best position to be partnered-with; (c) carrying cost rounds to zero, so no scenario strands the investment badly. The play is to be canonical *before* they decide.

**Risk: Congress.gov API dependency.** Healthy (5,000 req/hr, actively developed). The architecture already contains the mitigation: the API is touched only by the nightly sync; the runtime serves baked JSON. An outage degrades *freshness*, never *availability* — with an honest, drifting `as_of` date. `unitedstates/congress-legislators` covers the member spine independently. Residual: the Census geocoder vintage pin (119th → 120th bump), TheNewsAPI (optional; degrades to nothing by design).

**Risk: AllSides CC BY-NC.** Lean labels excluded from the MCP and from anything commercial. Standing flag until relicensed or re-sourced.

## 7. How the MCP feeds institutional revenue

The MCP is **top-of-funnel** for the two things institutions will actually pay for: white-label embeds and high-volume keyed API access (see the embeds spec).

1. **Every MCP response advertises the source** — the envelope's `canonical_url` + attribution puts rostra.org in front of exactly the people who staff newsrooms, libraries, and Leagues, who ask assistants civic questions professionally.
2. **Canonical URLs land on pages with an institutional door**: one quiet footer line on bill pages and `/reps` — *"Newsroom or library? Embed this — free with attribution."* → `/for-newsrooms` (Datawrapper's exact on-ramp).
3. **Free embed with attribution** is the marketing; the attribution link compounds.
4. **The paid conversion is white-label + volume**: the dormant `X-Rostra-Key` activates as the keyed API/MCP tier at 10× limits with an uptime commitment; billing = Stripe off-platform against the key (the only documented MCP-monetization pattern that works). Institutional keys are constitutionally clean.
5. **Sell through networks, not logos.** The MCP's role in a LION/INN/Press Forward pitch is **credential, not product**: "already the congressional connector in the Claude directory; cited by major assistants; here's the traffic."
6. **Lead capture without irony:** `/for-newsrooms` has a key-request form (org name + work email — institutional data, allowed). MCP-referred traffic to that form is the one conversion metric that matters for the Feb 2027 gate.

**Sequencing:** MCP + playbook after the citizen-site launch; the embed widget is a separate build gated on its own demand test. If MCP traffic is real by October, the widget rides the fall window; if not, the widget decision gets made with evidence instead of hope.

## 8. Decision gate

**~Feb 2027** (90 days post-midterms): review MCP call volume, directory-referred site traffic, embed/key inquiries, and any philanthropy conversations. Growing → activate the institutional lane properly (keys, Stripe, SLA) and pursue a network deal. Flat → the server stays up as a near-zero-cost public good and canonical citation asset. Either way the decoded bilingual corpus, the ZIP/office data spine, and the registry placements remain durable assets.
